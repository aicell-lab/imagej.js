ImageJ.JS: bioimage analysis as a versioned, AI-addressable URL

Wei Ouyang1,✉ · [Author 2]2 · [Author 3]3 · [Author 4]1

Bioimage analysis results depend on which installer, which plugin version, and which operating system a user last updated — a fragility that has become a reproducibility tax on the field. We show that a complete Fiji environment can run deterministically inside a web browser as WebAssembly, pinned by a versioned URL, and addressed by scripts or AI agents through Model Context Protocol. A published figure is re-runnable from a single link years later, on any device, with no installation and no data upload.

A bioimage analysis result depends on more than the image and the intent: it depends on the operating system, the Fiji version, the plugin checkout, and the month the installer was last touched. In a sample of 200 open-access microscopy papers published 2022–2024, [48]% used Fiji or ImageJ as the primary analysis tool (Fig. 1)4–6, and the result of a published macro is, in our hands, not reliably reproducible across updates of that installation (Fig. 2b, third candidate). The methodological contribution of this paper is not a new segmenter; it is the observation that if the analysis environment itself is content- addressable — a URL, not an installer — reproducibility collapses to the same primitive a reader already uses to access the paper.

ImageJ.JS runs the full Fiji4 application in a browser tab (Fig. 2a). CheerpJ7 ahead-of-time-compiles the Fiji JVM bytecode to WebAssembly on page load; image I/O, the macro interpreter, the ROI manager, and the plugin search path are served from a static bundle and executed client-side. The plugin-compatibility matrix in Fig 1b summarises which of the 40 most-cited Fiji plugins survive the port: pure-Java plugins (MorphoLibJ, TrackMate, BoneJ, MIJ, 3D ImageJ Suite) run unmodified and at interactive latency; plugins that shell out to a native binary (Weka v3 native, BigStitcher GPU, Imaris File Reader) do not. No image content, filename, or parameter is transmitted to any server by default; the browser sandbox is the audit boundary, and the property is structurally enforced, not policy-declared.

Reproducibility in ImageJ.JS derives from two structural properties, not from a best-practices guideline. The version of the tool in an analysis is a URL. A link to https://ij.aicell.io/?v=v1.0-paper loads precisely the build the authors used — the git-tagged Fiji JVM bytecode, the macro interpreter, and the plugin bundle, pinned byte-for-byte — so when upstream Fiji or a plugin changes months later the published link still renders the original analysis. Analyses are deterministic across browsers because CheerpJ executes the same bytecode on every platform, eliminating JVM-version-, OS-, and architecture-dependent drift. We packaged these properties into a replay corpus that re-runs published figures from three microscopy papers (Fig. 2b); each entry is scored on three independent axes — ACQUIRE (inputs match), EXECUTE (macro runs), MATCH (outputs agree within tolerance). In the Week-1 pilot, 2 of 3 candidates match on all three; the third (a watershed segmentation from a 2018 microscopy paper) matches on ACQUIRE and EXECUTE but fails MATCH because the authors' native Fiji install had pulled a 2020 point-release of the MorphoLibJ watershed implementation that reversed the default connectivity from 4- to 8-connectivity. The original 2018 bytecode, pinned in ImageJ.JS, reproduces the figure exactly. This is the argument for client-side pinning, not against it.

Three field deployments exercise ImageJ.JS where conventional bioimage tools are structurally excluded (Fig. 2c). In an undergraduate cell-biology course at [Partner university], 15 students analysed provided image sets on school-issued Chromebooks with no local install; standard course tasks (cell counting, colony morphology, fluorescence thresholding) ran at interactive latency, and end-of-course task completion was [Y] / [N] versus [Z] / [N] in the prior year's native-Fiji cohort. In a diagnostic histology workflow at [Partner hospital] under IRB [IRB-number], ImageJ.JS was installed on an air-gapped workstation; no slide pixel left the device. In a cross-institution collaboration demonstration, two investigators on different continents measured the same image dataset in real-time via a shared browser session, with per-user provenance attributed to authenticated Hypha8 identities.

A URL-addressable analysis environment is machine-addressable as well as human-addressable: the same endpoint that loads ImageJ.JS in a browser tab exposes runMacro, takeScreenshot, getRoisAsGeoJson, and executeJavaScript as Hypha8 remote-procedure-call methods and, via a URL rewrite, as Model Context Protocol18 tools addressable by any compliant AI agent (Fig. 3b). This is, to our knowledge, the first published bioimage analysis environment directly callable by an LLM-based scientific assistant without wrapper glue or a separate service. We used this capability to evaluate ImageJ.JS against foundation-model segmenters on a 30-task long-tail benchmark drawn from nine microscopy subdomains under-represented in training corpora (Fig. 3a). Four foundation models (SAM3, Cellpose1, StarDist2, CellSAM9) at default parameters matched the best expert segmentation on [Z] / 30 tasks; on the remainder they systematically over-segmented soft boundaries (plant root tips, marine plankton) or failed to detect sparse small objects (soil microbial, citizen-science trail-cam). Experts using ImageJ.JS completed all 30 by combining the Trainable Weka classifier, the MorphoLibJ watershed, and two task-specific macros per subdomain. The two approaches are complementary, not competing: for in-regime tasks we recommend the foundation model; ImageJ.JS is the general-purpose tool for what foundation models have not yet been trained on.

ImageJ.JS does not train models or accelerate GPU inference; for tasks a foundation model already solves, we recommend using it. What we contribute is methodological: an analysis environment whose version is a URL, whose execution is deterministic across devices, and whose interface is addressable by both a human mouse-click and an AI-agent tool call. Comparable browser-based tools serve orthogonal needs — WebKnossos19 for large-volume connectomics annotation, OMERO.web16 for data sharing, CellProfiler Analyst13 for high-throughput batch analysis — but none runs the general-purpose Fiji ecosystem client-side with pinned-URL reproducibility. The reproducibility primitive we describe — an analysis environment as a content-addressable URL — is independent of Fiji and transfers to any Java-, JavaScript-, or WebAssembly- shippable analysis stack.
