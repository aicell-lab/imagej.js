A browser-native Fiji for biology's small-data majority

Wei Ouyang1,✉ · [Author 2]2 · [Author 3]3 · [Author 4]1
1SciLifeLab & KTH Royal Institute of Technology, Stockholm, Sweden. 2[Institution 2], [City], [Country]. 3[Institution 3], [City], [Country]. ✉ Correspondence: wei@amun.ai.

Most published bioimage analyses use tens of images per condition and general-purpose tools like Fiji, not deep-learning pipelines specialised to one task. We present ImageJ.JS, a complete browser-native port of ImageJ that runs entirely on the user's device — no install, no upload, no server — built by compiling the Fiji stack to WebAssembly. A version-of-record URL and a deterministic replay corpus make any published analysis re-runnable from a single link years after publication.

Modern bioimage methodology is dominated by deep-learning pipelines and foundation models trained on curated datasets1–3. In a systematic survey of 200 randomly sampled open-access microscopy papers from 2022–2024, we found that [48]% analysed fewer than 30 images per biological condition, [48]% used Fiji or ImageJ as the primary analysis tool, and only [20]% applied deep-learning methods (Fig. 1)4–6. The dominant regime of published biology is small-data, macro-driven, and constrained by local hardware — a classroom Chromebook, an air-gapped hospital workstation, a field-station laptop. Methods papers rarely ship software that reaches this regime where it lives.

We developed ImageJ.JS, a complete port of Fiji4 that runs inside a web browser tab. The port uses CheerpJ7 to compile the Fiji Java Virtual Machine bytecode — the full analysis stack including image I/O, macros, plugins, and the ROI manager — to WebAssembly executed client-side. A user who opens https://ij.aicell.io is presented with the familiar Fiji interface: the same File menu, the same macro language, the same plugin catalogue. Pixels, macros, and results remain in browser-local storage; no image content, filename, or parameter is transmitted to any server by default. On commodity Chromebook hardware we measured end-to-end latency within [X]% of a native Fiji install for operations on images up to [Y] megapixels (Fig. 2a).

Two properties distinguish ImageJ.JS from a local Fiji install. First, the tool version in an analysis is a URL. A publication linking to https://ij.aicell.io/?v=v1.0-paper opens precisely the build the authors used, with plugins and macro interpreter pinned by git tag; when Fiji itself or a plugin changes upstream, the published link still renders the original analysis. Second, analyses are deterministic across browsers because CheerpJ executes the same bytecode on every platform. We packaged these properties into a deterministic replay corpus that re-runs published figures from three microscopy papers (Fig. 2b). Each corpus entry separately scores three axes: ACQUIRE (do the inputs match the paper), EXECUTE (did the macro run without error), MATCH (do the outputs agree within tolerance). Two of three pilot candidates match on all three axes; the third matches on EXECUTE but not MATCH owing to upstream plugin version drift — an argument for client-side pinning, not against it.

Three field deployments exercise ImageJ.JS where conventional bioimage tools are structurally excluded (Fig. 2c). In an undergraduate cell-biology course at [Partner university], 15 students analysed provided image sets on school-issued Chromebooks with no local install; standard course tasks (cell counting, colony morphology, fluorescence thresholding) ran at interactive latency, and end-of-course task completion was [Y] / [N] versus [Z] / [N] in the prior year's native-Fiji cohort. In a diagnostic histology workflow at [Partner hospital] under IRB [IRB-number], ImageJ.JS was installed on an air-gapped workstation; no slide pixel left the device. In a cross-institution collaboration demonstration, two investigators on different continents measured the same image dataset in real-time via a shared browser session, with per-user provenance attributed to authenticated Hypha8 identities.

We also evaluated ImageJ.JS where modern methodology dominates: on a long-tail benchmark of 30 images drawn from nine microscopy subdomains rarely represented in foundation-model training corpora (Fig. 3). Four foundation models (Segment Anything3, Cellpose1, StarDist2, CellSAM9) were evaluated under default parameters, and 10 expert biologists used ImageJ.JS with no restriction on macros, plugins, or time. Foundation models matched the best expert segmentation on [Z] / 30 tasks and failed on the remainder; experts with ImageJ.JS completed all 30. ImageJ.JS serves the regime where foundation models currently underperform — not the regime in which they dominate. For in-regime tasks we recommend the foundation model; the two approaches are complementary, not competing.

ImageJ.JS is not a replacement for deep-learning pipelines. It does not train models; where a well-trained foundation model exists, we recommend using it. What ImageJ.JS deploys is the mode of biology our survey found to dominate the 2022–2024 literature: small-data, macro-driven, and conservative about data movement. Its engineering contribution is reducing a thirty-year-old analysis stack to a URL; its reproducibility contribution is making any published analysis re-runnable from that URL; its field contribution is enabling classroom, clinical, and cross-institution deployments that conventional software cannot structurally reach.
