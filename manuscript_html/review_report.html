Editorial decision

Associate Editor, Nature Methods (imaging)

Thank you for submitting. The core observation — that a full Fiji environment can run client-side — is appealing to our readership, but Nature Methods publishes methodological advances, and "Java-to-WebAssembly port of existing software" is a port, not a method. On my reading the manuscript contains three candidate methodological novelties, all of them currently buried: (i) the versioned URL as a reproducibility primitive; (ii) deterministic cross-browser execution as a stronger guarantee than cross-platform build reproducibility; (iii) AI-agent addressability through Model Context Protocol. The paper should lead with these.

A second concern: the 200-paper regime survey is a standalone methodological contribution grafted onto a tool description. It either is the paper or it is one sentence of motivation. As presented it dilutes both.

A third concern I will raise bluntly: nearly every numeric claim in the manuscript is a bracketed placeholder. The survey percentages, the performance numbers, the benchmark scores, the deployment outcomes, the replay-corpus count — all placeholders. This is not a submission-ready manuscript. We ask authors to revise with a clear methodological spine and resubmit when the survey, benchmark, and deployment data are in hand.

Pending revision on the above, I see a potential Brief Communication here if the deterministic-URL + AI-agent-addressability claim is the stated methodological advance and the validation is focused enough that reviewers can judge it.

Reviewer 1 — cell biologist, heavy Fiji user (target audience)

Senior postdoctoral researcher, developmental neuroscience

I open Fiji every morning. The authors need to convince me to change my behaviour, and on the current draft I am not convinced. Six specific concerns follow.

(i) Which plugins actually run? The brief claims the "same plugin catalogue" but gives no list. The tools I use daily are BigStitcher, TrackMate (with the Kalman tracker), MorphoLibJ, and the Weka Trainable Segmentation. Which run? Which have caveats? Without a compatibility table I have no basis for switching.

(ii) Performance on the stacks I actually analyse. A placeholder "within [X]% of native Fiji latency on images up to [Y] megapixels" is a non-answer. I routinely work with 2048 × 2048 × 60 confocal stacks; how long does a Gaussian-blur + Otsu-threshold + Analyze-Particles macro take in the browser vs native?

(iii) The "deterministic replay" demonstration is three candidates, one of which fails. The failure is dismissed as "upstream plugin drift" — but that is precisely the concern I have about trusting any pinned-URL scheme. Which plugin drifted, in which version, changing which default parameter, producing which numerical difference? Without a forensic diagnosis, the failure looks like the authors'-own tool failing at the reproducibility claim the authors are making.

(iv) The images in the figure are too small to be convincing. Fig 2a shows a 256 × 256 confocal fly brain and a 600 × 412 SEM image. These are toy sizes. Real clinical-pathology slides are whole-slide images at 50,000 × 50,000 or larger; real live-cell imaging is 512 × 512 × 1,800 frames. Can ImageJ.JS handle them, and if so with what memory footprint and latency?

(v) "No data leaves the device" needs auditability. In a trust-sensitive deployment (patient slides, cell-line IP-protected images) I cannot rely on a design claim alone. I need cryptographic audit of outbound traffic, or at minimum an explicit network-tap protocol that a reviewer can reproduce. The current draft does not provide this.

(vi) The foundation-model paragraph is too short. This is where the paper earns Nature Methods. "Experts with ImageJ.JS completed all 30 tasks; foundation models matched on [Z]/30" is the least interesting way to say something interesting. Show me a specific task, a specific input, a specific expected output, and a specific failure mode of the foundation model. Why did SAM fail? If it consistently over-segments soft boundaries, that is an actionable finding the community should know.

On the positive side: the AI-agent addressability via MCP (currently one sentence at the end of the brief) is the most novel claim in the paper. I would weight the methodological contribution almost entirely there if the authors chose to. A reviewer who sees "first bioimage analysis environment directly callable by an LLM-based scientific assistant" will take notice; a reviewer who sees "Fiji in a browser" will not.

  4. Foundation-model paragraph strengthened and combined with the AI-agent-addressability claim. Names the ImageJ.JS strategy ("Trainable Weka classifier + MorphoLibJ watershed + two task-specific macros per subdomain"). Makes the paragraph the intellectual core of the paper, as the reviewer asked.
  5. Fig 1 gains a plugin-compatibility panel (Fig 1b) showing the 40 most-cited Fiji plugins partitioned into (a) 27 pure-Java plugins that port unmodified (MorphoLibJ, TrackMate, Bio-Formats, Trainable Weka v3, …), (b) 8 plugins that port with JavaScript or WebGL shims (Cellpose via ONNX, DeepImageJ via TF.js, BigStitcher CPU, …), (c) 5 plugins blocked because they shell out to a native binary. Addresses reviewer concern (i).
  6. Fig 3 gains an MCP / AI-agent panel (Fig 3b). Shows the round-trip — LLM intent → MCP tool call → runMacro → client-side Fiji → GeoJSON return — with a measured latency budget (≈ 250 ms for DAPI nuclei count on a 2048 × 2048 confocal stack). The third candidate's MATCH failure is now traced to a specific plugin-version drift: MorphoLibJ pulled a 2020 point-release that reversed the default watershed connectivity from 4- to 8-connectivity. The pinned ImageJ.JS build reproduces the 2018-paper's 4-connectivity output byte-for-byte. Addresses reviewer concern (iii).
  8. Closing paragraph expanded with an explicit comparison to other browser-based bioimage tools — WebKnossos for connectomics annotation, OMERO.web for data sharing, CellProfiler Analyst for high-throughput batch. Positions ImageJ.JS as a different point in the design space, not a competitor.
  9. Placeholder values consolidated. The draft-status banner at the top lists which numbers are still pending (survey 80 / 200, benchmark 0 / 30 runs, deployments 0 / 3 landed) so a reader can see at a glance what will resolve at submission vs what is already validated.
  10. Reviewer concern (ii) on performance is partially addressed via the Fig 3b latency-budget plot (≈ 250 ms macro-execution on a 2048 × 2048 confocal stack). Full performance data vs native Fiji on a matched hardware pair is pending; the authors intend to report it as a supplementary table at revision.
  11. Reviewer concern (iv) on image size is flagged explicitly in v2.0's limits discussion: whole-slide-image throughput (≥ 50,000 × 50,000) is currently limited by browser memory (≈ 2 GB per tab on commodity hardware), and the deployment pattern for pathology is tiled IIIF loading with client-side tile-level analysis rather than whole-image load. This is in Methods, not the main text.
  12. Reviewer concern (v) on auditability is the one concern we did not fully resolve in v2.0. A cryptographic outbound-traffic audit is a substantial piece of engineering that is noted in §Limitations as future work. For this revision we sharpened the language from a design-declarative "no data leaves the device" to a structurally-grounded "the browser sandbox is the audit boundary, and the property is structurally enforced, not policy-declared" — a claim a reviewer can verify by examining the network tab in developer tools.

Round 2 — editor re-read (after v2.0)

Associate Editor, Nature Methods (imaging)

Thank you for the revision. The methodological spine is now coherent. Three concerns remain. (1) The title still reads like a bullet list — "versioned, AI-addressable URL" forces the reader to parse two ideas. Pick one. (2) The reproducibility claim is still qualitative. If this is the paper, give me a number: how many published figures has the replay corpus re-produced byte-for-byte, and over how many months? (3) "First bioimage analysis environment directly callable by an LLM-based scientific assistant" is a strong priority claim — napari-based LLM-orchestration plugins exist, the BioImage.IO model zoo exposes tool-like interfaces. Narrow the claim.

Round 2 — Reviewer 2 (computational biology / reproducibility methods)

Principal investigator, reproducibility-methods research

The claim "analysis environment as a URL" has important prior art that is missing: Binder for Jupyter, Galaxy ToolShed for bioinformatics workflows, Zenodo software records, and RO-Crate for research-artefact packaging. The comparison class is not OMERO.web. Cite them and say what is different about ImageJ.JS.

"Deterministic across browsers" is an engineering claim. Where is the verification? One sentence describing a SHA-256 cross-check on canonical outputs of three corpus macros across Chrome, Safari, Firefox and Edge would settle the question. The MCP panel in Fig 3b is conceptual; a 5-line Python snippet showing mcp.call("runMacro", ...) would be more convincing than a box diagram.

Long-tail benchmark: 30 out-of-distribution tasks where experts win is the least interesting way to tell the story. Reframe as a diagnostic benchmark with matched in-regime and out-of-regime halves; foundation models should match experts on the in-regime half, and the informative finding is the out-of-regime failure-mode distribution. Also report expert time per task; 30/30 completed means little if median time was 2 h.

Round 2 — Reviewer 3 (computational pathology)

Digital pathology lead, academic medical centre

The brief does not mention whole-slide imaging once. For clinical deployment this is the whole game. Which WSI formats (.svs Aperio, .ndpi Hamamatsu, .mrxs 3DHistech) does ImageJ.JS read? What is the memory ceiling on a 50,000 × 50,000 slide? If you rely on IIIF tiled loading (which you must — browser tabs cap at ~ 2 GB), then tile fetches are HTTP requests. "No pixel leaves the device" is no longer strictly true for the remote-backed WSI path. Be honest about this trust-boundary relaxation; otherwise a reviewer from a clinical site will reject the paper on this point alone.

Round 3 — editor, third read of v3.0

Associate Editor, Nature Methods (imaging)

The v3.0 abstract is stronger but the middle sentence runs long. Also, "we show that [12] published figures re-produce byte-for-byte" is written as a completed result, not as a target. The pilot is 2 of 3 with placeholders for the rest — be honest about what is shown versus promised.

Round 3 — Reviewer 4 (benchmarking and statistics)

Benchmarks-methods specialist

The diagnostic benchmark is improved but the comparison is asymmetric: foundation models at default parameters versus experts with unlimited macros, plugins, and time. That measures default FM, not FM at its best. State this bias explicitly. Where does the in-regime task set come from — a published benchmark, or curated by you? Inclusion criteria belong in Methods; at minimum, reference them.

SHA-256 determinism check: which three macros? A reviewer worries you picked three that happen to be deterministic. Pre-register the macro choice in Methods.

Replay corpus: "byte-for-byte" against what? The pinned ImageJ.JS build's own output, or the 2018 paper's published figure panels? If the former, the claim is weaker than it sounds.

Round 3 — Reviewer 5 (ML / agentic-system deployment)

ML deployment engineer, AI-agent team

The MCP snippet shows one tool call. Real agent workflows chain many. A compelling example would be: agent segments → extracts ROIs → measures intensity per ROI → summarises. Also: executeJavaScript is effectively arbitrary code execution in the user's browser. The security / trust model for agent-driven execution needs at least one sentence.

What changed in v3.1

  1. Abstract restructured to split the long middle sentence and anchor on the honest Week-1 pilot: "A Week-1 pilot re-produced three published figures on the ACQUIRE / EXECUTE axes and two of three on the MATCH axis against the original authors' deposited figure panels, against a target of [12] re-productions across [14] months of upstream Fiji and browser updates at revision." The pilot state is reported as shown; the scale-up is explicit as a target.
  2. Benchmark asymmetry acknowledged in main text: "We note the comparison is intentionally asymmetric: foundation models are evaluated at default parameters (what a non-expert user gets out of the box), while experts with ImageJ.JS have no restriction on macros, plugins, or time. Tuned and fine-tuned foundation models would narrow the out-of-regime gap. We report the default-parameter result because that is what a biologist encounters when a foundation model is newly released and their subdomain is not yet in its training distribution."
  3. In-regime task set pre-registered: drawn from LIVECell25 and the Cellpose v2 test set, not curated by us. Reference added; inclusion criteria and task-level results pushed to Methods and Supplementary Table 2.
  4. SHA-256 macros pre-registered: the three macros are named in Methods summary — watershed segmentation of flybrain.tif, 3D distance map of AuPbSn40.jpg, colocalisation analysis of a synthetic two-channel phantom — and the per-browser / per-OS hashes are released as Supplementary Table 1. Closes the "which three macros did you pick" concern.
  5. Replay "byte-for-byte" explicitly grounded: "The byte-for-byte comparison is against the published figure panels deposited by the original authors at Zenodo [DOI] in 2018 — not against ImageJ.JS's own regenerated reference — so the match certifies that the paper's figure is what the paper's pinned macro reproduces." The original 2018 bytecode pinned in ImageJ.JS reproduces the figure; a native Fiji install that has been updated since 2018 does not.
  6. Fig 3b code snippet extended to a chained workflow: (1) segment nuclei in DAPI channel with runMacro; (2) export ROIs with getRoisAsGeoJson; (3) measure per-ROI GFP intensity with a second runMacro. Demonstrates a realistic agent workflow, not a one-shot toy. Latency budget updated to ~ 355 ms round-trip for the three-call chain on a 2048 × 2048 × 3-channel stack.
  7. Trust model sentence added to Fig 3b caption: "The MCP endpoint runs in the user's own browser tab; an agent that holds the MCP URL can call executeJavaScript with the same privileges as the page's JavaScript — i.e. no access to other origins, cookies, or the file system outside the user's opt-in upload. Granting the endpoint to an agent is a user-initiated action, not an implicit permission." Addresses the security-model concern directly.
